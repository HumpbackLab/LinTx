# LinTx显示技术手册

## 1. 这份文档讲什么

这份文档只做两件事：
- 介绍 LinTx 当前会碰到的显示概念。
- 说明 SG2002 实物上现在的显示架构。

重点关注这些内容：
- `fb`
- `VPSS / VENC / VO`
- 常见颜色格式
- `LVGL` 后端
- 启动 Logo、LinTx UI、媒体 sample 之间的关系

## 2. 当前显示全貌

把当前平台的显示过程看成三段就够了：

1. `启动阶段`
   由引导链和 SDK 负责，先把 Logo 或基础画面送到屏幕，避免黑屏。

2. `应用阶段`
   进入 LinTx 后，主界面走 `LVGL + fbdev + /dev/fb0`。

3. `媒体阶段`
   你在 SDK 里加入的板级 H.264/H.265 播放程序 `sample_vdec_bind_vo` 维护了两条实际链路：主链是 `VDEC -> VPSS -> VO`，回退链是 `VDEC -> VPSS -> userspace blit -> /dev/fb0`。

这三段都和显示有关，但不是同一套东西：
- Logo 解决的是“开机先亮起来”。
- LinTx UI 解决的是“应用界面怎么显示和交互”。
- 媒体 sample 解决的是“视频帧怎么处理、怎么上屏”。

## 3. Framebuffer（fb）是什么

`framebuffer` 可以理解成一块供用户空间直接写入的屏幕缓冲区。应用只要按分辨率、步长和颜色格式把像素写进去，屏幕就会显示对应内容。

对 LinTx 来说，`fb0` 适合做主 UI，原因很直接：
- 路径短。
- 依赖少。
- 适合菜单、状态页、触摸界面这类 2D UI。

LinTx 当前板端主路径是：
`LinTx 应用逻辑 -> LVGL -> fbdev 后端 -> /dev/fb0 -> 面板`

这条链的特点是：
- 由应用自己画屏。
- 更接近 GUI，而不是视频播放链路。
- 显示和触摸输入放在同一套应用后端里处理。

在当前实现里，fb 路径还额外处理了几件板级适配工作：
- 屏幕方向。
- 红蓝通道顺序。
- 翻页刷新。
- 触摸坐标和屏幕方向的一致性。

## 4. VPSS / VENC / VO 是什么

### 4.1 VPSS

`VPSS` 是图像处理模块，可以把上游送来的图像整理成下游更容易使用的形式。

它主要负责：
- 裁剪
- 缩放
- 旋转
- 镜像
- 像素格式转换

在当前平台上，VPSS 常处在媒体链中间。无论图像来自相机还是解码器，通常都要先经过它，再决定送显示还是送编码。

### 4.2 VENC

`VENC` 是视频编码模块，负责把图像帧压成码流，比如 H.264、H.265、JPEG。

它主要用于：
- 推流
- 录像
- 生成压缩视频输出

它不是主显示模块。对当前工程来说，VENC 更像“媒体输出能力”，不是 LinTx 主界面的一部分。

### 4.3 VO

`VO` 是视频输出模块，负责把媒体链中的图像帧送到显示设备。

它和 `fb0` 都能把内容送到屏幕，但方式不同：
- `fb0` 是应用直接写屏。
- `VO` 是媒体链把图像帧送到显示输出。

VO 更适合这些场景：
- 视频播放
- 解码后上屏
- 需要和 VPSS 连续配合的媒体输出

### 4.4 在当前工程里的位置

当前工程里可以把两条链分开理解：

- `LinTx UI 链`：`LVGL -> fbdev -> fb0`
- `媒体播放链`：以 `sample_vdec_bind_vo` 为代表，主链是 `VDEC -> VPSS -> VO`，回退链是 `VDEC -> VPSS -> userspace blit -> fb0`
- `媒体输出链`：需要生成码流时再进入 `VENC`

前者服务界面，后两者服务视频播放和媒体输出。

## 5. 颜色格式为什么重要

显示系统里最常见的问题之一，不是“画不出来”，而是“颜色不对”或者“格式不对”。原因通常是上下游看到的不是同一种像素数据。

LinTx 相关的显示链里，最常见的是三类格式。

### 5.1 RGB565

这是嵌入式 UI 常见的 16 位格式：
- 红 5 位
- 绿 6 位
- 蓝 5 位

它占用小，适合界面绘制。LinTx 当前的 LVGL 侧主要站在这类 RGB 界面的视角工作。

### 5.2 ARGB8888 / 32bpp RGB

这是 32 位格式，颜色表达更直接，但内存和带宽开销更大。

它常见于：
- 32 位 framebuffer 写屏
- 需要和系统原生像素布局对齐的路径

### 5.3 NV21

这是媒体链中很常见的 YUV420 半平面格式。

它适合：
- 视频处理
- 硬件缩放
- 硬件旋转
- 编码
- VO 输出

在 SDK 的 sample 和媒体链里，NV21 是很常见的中间格式。

### 5.4 当前工程里的实际分工

可以把当前格式分工简单理解成：
- `LinTx UI` 更接近 `RGB`
- `媒体 sample` 更接近 `NV21/YUV`

因此当前系统里经常会出现：
- 格式转换
- 旋转
- 通道顺序调整

这不是额外设计，而是 UI 世界和媒体世界交界时的正常现象。

## 6. LVGL 后端怎么理解

`LVGL` 是当前 LinTx UI 的界面库，负责组织页面、控件和刷新逻辑。

但 LVGL 不直接决定内容画到哪里，它需要后端承接输出。

当前工程里可以看到三类后端：
- `terminal`：最小化调试
- `sdl`：桌面调试
- `fbdev`：板端真实显示

所以可以把 LVGL 和后端的关系理解成：
- LVGL 负责生成界面
- 后端负责把界面送到具体设备

在板端，当前实际使用的是 `fbdev`，也就是把 LVGL 的输出写到 `fb0`。

## 7. 当前实物架构

如果只看今天这块板子，显示架构可以直接理解成下面这样：

### 7.1 启动 Logo

启动阶段先由引导链和 SDK 把 Logo 或基础画面送到屏幕。这一阶段的目标只有一个：设备上电后尽快有画面。

### 7.2 LinTx UI

进入 LinTx 后，主界面通过 `LVGL + fbdev + fb0` 显示到屏幕。

这部分负责：
- 菜单
- 页面
- 状态显示
- 触摸交互

这是当前实物上的主界面链路。

### 7.3 媒体 sample

在进行H264解码测试的时候，在SDK的fork中加入了 `sample_vdec_bind_vo`这个例程。

它当前维护两条实际播放路径：
- 主路径：`VDEC -> VPSS -> VO`
- 回退路径：`VDEC -> VPSS -> userspace blit -> /dev/fb0`

主路径面向板载面板播放，回退路径用于 VO 不稳定时的对比和兜底。

这部分主要用于：
- 本地 `.264` / `.265` 码流播放
- 解码上屏验证
- 旋转、缩放、格式转换验证
- 后续流式视频接入前的板级播放基础

### 7.4 三者之间的关系

三者不是互相替代，而是分工不同：
- `Logo` 负责开机早期显示
- `LinTx UI` 负责应用界面
- `媒体 sample` 负责板级解码播放和媒体能力验证

所以当前实物上并不是“只有一套显示系统”，而是“同一块屏幕上有几种不同的显示路径”。

## 8. 一句话总结

对当前 LinTx 和 SG2002 实物来说，最实用的理解方式是：

- 日常界面显示，主要看 `LVGL + fbdev + fb0`。
- 视频和媒体能力，主要看 `VPSS / VENC / VO`。
- 开机到应用的显示过程，可以理解为 `Logo -> LinTx UI -> 媒体链能力`。
